Gettin' Dynamic with ASP

July 12, 1999.

Whoa! The hubbub is finally over... Flash 4 is here and everyone's scrambling to be the first on the block with draggable whatsamadoovies and editable thingamejigs. That's great - but if you ask me the most exciting new possibility in the Flash realm is client/server applications and dynamic data access.

"Dear Aramis,

Is there a way to incorporate ASP data access with Flash? If not are there other ways to do data access within flash? Thanks.

- Andy"

Hi Andy,

ASP, or Active Server Pages, is just one technology you can use to supply Flash with dynamic data. Also in the HTML preprocessor family are PHP and ColdFusion. Then there's CGI scripts, commonly written in Perl, or static text files, and of course Macromedia Generator - but that's a slightly different kettle of fish.

So how the heck can this happen?

The Magical Loading Variable

Those hipster briefs at Macromedia have done us all a big favor and added an excellent new option to the Load Movie action called Load Variables. If you're completely in the dark about Flash 4 variables, check out our tutorial as a primer.

In a nutshell, variables are a big deal when it comes to building Flash applications. You can use variables to store everything from a person's name or telephone number, to True/False data and other cryptic stuff. By getting variable data from an outside source, an online database for example, you can either directly display the info in a text field, or manipulate it further using inbuilt ActionScript functions.

Tell Me More...

Always trying to take my own advice, I decided to consult the mystical Help files and see what they had to say on the subject of obtaining data from remote files. Here's the gospel according to the big Macs:

"Use the Load Variables statement, which is one of the parameters of Load Movie, to read data from a text file or text generated by a CGI script, and set the values for variables in a movie or movie clip. For example, if a user submits an order form, you might want a confirmation screen to display an order number gathered from a file on a remote server.

With Load Variables, the text at the URL must be in the standard MIME format application/x-www-urlformencoded (a standard format used by CGI scripts). Any number of variables can be specified. For example, this phrase defines several variables:

company=Macromedia&address=600+Townsend&city=San+Francisco&zip=94103

  • URL specifies an absolute or relative URL for the remote file.

  • Location specifies a level or target to receive the variables. In the Flash Player, movie files are assigned a number according to the order in which they were loaded. The movie that was loaded first is loaded at the bottom level, level 0. The movie in level 0 sets the frame rate, background color, and frame size for all other loaded movies. Movies are then stacked in higher numbered levels above the movie in level 0.

  • Variables determines whether to use GET or POST to pass variables, or to pass no variables at all. This can be used to send variables to a CGI script that generates a SWF file as its CGI output. The GET method appends the variables to the end of the URL, and is used when a small number of variables are involved. The POST method sends the variables in a separate header, and is capable of sending much longer strings. See Setting and identifying variables.

If you specify either GET or POST with the Load Variables statement, Flash sends the variables to
the web server, and then interprets the server's response to update the variables in the movie.

Use the File > Publish Preview option to test the loading of variables. (An active network connection must exist for this to work.)"

Err, say that again?

Help files are wonderful, they tell you everything you need to know in a way that requires a variety of illicit substances to understand. Here's the highlights in Aramis-speak...

Figure out your script's URL
Hopefully this should be a no-brainer. Let's say you have an ASP file that loads data from your SQL database and returns a set of results, the URL in your Load Variables action would look something like http://www.mycompany.com/thatscript.asp

Wanna send the script some stuff? Sure thing...
Obviously this whole loading data thing works both ways. If you want to get some info from the database, you'll need to tell it what you're looking for. Flash just takes all the variables currently available in your movie and passes them along in a big lump. All you have to decide is whether to send using GET or POST methods... and that depends on what method your page is looking for.

For instance in ASP, Request.Form("variablename") deals with variables sent using POST and Request.QueryString("variablename") extracts data sent using the GET method.

My advice is try to use POST where possible because GET is restricted to about 250 characters, no good for beefy data.

Have your script pump out the right stuff
Flash can only understand data encoded in the standard MIME format "application/x-www-urlformencoded" - but that's not as mysterious as it sounds. In ASP, its as simple as using the Response.Write and URLEncode function to make sure everything's kosher.

The main concept to grasp here is that instead of pumping out HTML code as you'd normally want to, your script just creates a big long string of name/value pairs in a format like -

company=Macromedia&address=600+Townsend&city=San+Francisco&zip=94103

Stick it all in the correct location
Once you've loaded in all this data, its gotta go somewhere right? If you've used the Load Movie action before you'll be familiar with the concept of Levels. Basically the idea here is that if you want to keep different values of the same name separated (to prevent overwrites) you can load external data into a particular Layer. Then when you want to retrieve it later you just append "_layerX" to the front of your variable identifier, where X is the desired Layer number.

Good Guineas! That's pretty cool.

Sure is! And it's easy peasy so don't be getting all confused and stuff :-) Happy Flashing and have a great week.

Aramis
